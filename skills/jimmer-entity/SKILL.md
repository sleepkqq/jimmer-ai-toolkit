# Jimmer Entity

Use for creating or changing Jimmer entities and their minimal repositories. Advanced mappings (@Formula, @IdView, @LogicalDeleted, @Embeddable, ...) live in `jimmer-advanced-mappings`.

## Workflow

1. Run project scan:

```bash
scripts/scan-project.sh /path/to/project
```

2. Read existing entities, mapped superclasses, repositories, and package style.
3. Ask only for missing requirements: entity purpose, fields, associations, natural key, audit/base fields.
4. Design entity using rules below.
5. Add reverse association sides in existing entities when required.
6. Create repository only if project uses repositories. Keep it empty unless current code calls a custom method.
7. Compile with:

```bash
scripts/compile.sh /path/to/project
```

## Entity Rules

- Jimmer entities are immutable interfaces generated by APT/KSP.
- Field order: `@Id` -> primary fields -> secondary fields -> audit/version -> associations last.
- Use built-in `UUIDIdGenerator`; never create custom id generators unless target project already mandates one. `GenerationType`: `IDENTITY`, `SEQUENCE`, `USER` (custom `UserIdGenerator`).
- Entity properties are non-null by default. Use `@Nullable` from `org.jetbrains.annotations` (Java) or `T?` (Kotlin) only for nullable fields. Do not add `@NotNull`.
- Do not add `@Column` for normal camelCase -> snake_case mapping.
- Add `@Table` only when table name is not default snake_case.
- Put `@JoinColumn` directly under association annotation. `@JoinColumn(foreignKeyType = ForeignKeyType.FAKE)` when no real DB FK exists.
- Use `@Key` for natural/business identity used by save/upsert. Multiple independent keys via groups: `@Key(group = "byName")`.
- Add `@KeyUniqueConstraint` when DB unique constraint exists or will be created. Postgres + nullable key columns: `@KeyUniqueConstraint(isNullNotDistinct = true)` with `NULLS NOT DISTINCT` constraint.
- `@OnDissociate` actions: `NONE` (use global default), `LAX` (skip check, DB decides), `CHECK` (forbid dissociation — error), `SET_NULL` (FK must be nullable), `DELETE`.
- Use `@Default("...")` for primitive defaults instead of magic zero handling.

## Base Entity Rule

If project already has `@MappedSuperclass`, ask whether new entity needs those fields. If yes, extend existing base. If none exists, ask before adding one.

Common pattern:

```java
@MappedSuperclass
public interface Auditable {
    Instant createdAt();
}

@MappedSuperclass
public interface Model extends Auditable {
    Instant updatedAt();
    @Version
    int version();
}
```

## Association Patterns

```java
@ManyToOne
@JoinColumn(name = "related_object_id")
RelatedObject relatedObject();

@OneToMany(mappedBy = "parent")
@OnDissociate(DissociateAction.DELETE)
List<Child> children();
```

- Reverse side: if new entity references existing entity, add mapped side only when navigation is needed or project convention requires it.
- One-to-one where target id equals own id: `@MapsId` (see `jimmer-advanced-mappings`).
- Middle-table many-to-many: `@ManyToMany` + `@JoinTable`; read-only shortcut through an association entity: `@ManyToManyView`.
- FK id without loading target: `@IdView`.

## Minimal Repository

```java
public interface DomainObjectRepository extends JRepository<DomainObject, UUID> {
}
```

Add a custom method only when code in current task directly calls it and `JRepository` built-ins cannot handle it.

## Anti-Patterns

- Speculative `findBy*` methods.
- Redundant `@Column(name = "some_field")` for default mapping.
- `@Key` on child side of shared `@OneToOne` without checking uniqueness semantics.
- Creating DTOs or migrations from entity task without user asking. Use `jimmer-dto` and `jimmer-migrations` separately.
